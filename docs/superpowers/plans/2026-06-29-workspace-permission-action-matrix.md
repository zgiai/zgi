# Workspace Permission Action Matrix

Date: 2026-06-29
Branch: `px/permission-system-overhaul`
Status: live audit checklist

This matrix records the current workspace asset permission contract after the fine-grained role refactor. It is scoped to frontend route/action guards and backend workspace permission checks. Published runtime authorization remains a separate surface contract.

## Rules

- `workspace.*` permissions are governance or membership signals. They are not selectable ordinary member permissions.
- Organization owner/admin and workspace owner/admin keep governance authority. Ordinary members receive only asset action permissions.
- `prompt.*`, `content_parse.*`, and `dashboard.*` permissions are retired from member templates.
- Page-visible groups are only for navigation, list, empty state, and direct-page access. Buttons and mutations must use one action permission group.
- Mixed agent/workflow containers may use the combined visible group only for the shared list. Once an `agent_type` is known, detail layouts and sub-routes must branch to `AGENT_PERMISSION_ACTIONS.page` or `WORKFLOW_PERMISSION_ACTIONS.page`.
- Coarse legacy codes such as `agent.manage`, `knowledge_base.manage`, `database.manage`, `database.data_edit`, `database.ai_query`, `file.manage`, `file.upload_create`, and `file.move_create` may remain as compatibility inputs, but UI actions should consume the expanded action permission. These aggregate codes are not selectable in editable member/template permission lists.

## Change Log

- 2026-06-29: Frontend scheduled-task management actions now use `isWorkspaceManager()` instead of generic workspace access, matching backend `workspace.manage` mutation checks without adding ordinary member `workspace.*` permissions.
- 2026-06-29: Agent/workflow detail layout now branches by loaded runtime type for page access. The shared list can still use the combined visible group, but detail shells no longer let AGENT visibility stand in for workflow visibility or vice versa.
- 2026-06-29: Backend API key management was aligned by runtime type: AGENT uses `agent.runtime_access.manage`, workflow/conversational workflow uses `workflow.runtime_access.manage`. The existing frontend API-key/docs detail route stays workflow-only per route-access contract.
- 2026-06-29: Workflow editor actions were split by exact permissions: save/canvas edit, run draft, stop run, node debug, publish, runtime access, and run history no longer share `workflow.update` or broad manage checks.
- 2026-06-29: Agent runtime surface settings and WebApp online/offline management now use `agent.runtime_access.manage` end-to-end. Runtime configuration edit remains `agent.runtime_config.manage`, while publishing remains `agent.publish`.
- 2026-06-29: Agent runtime editor now propagates `agent.runtime_config.manage` into a read-only prompt/orchestration state. Users without runtime config permission can still inspect the runtime setup, but prompt/model/resource/memory/experience controls are disabled or no-op guarded.
- 2026-06-29: Frontend selectable permission lists now hide compatibility-only aggregate codes `database.data_edit`, `database.ai_query`, `file.upload_create`, and `file.move_create`. Permission normalization expands those old codes into precise action permissions before saving, so editing an old snapshot does not silently lose equivalent capabilities.
- 2026-06-29: Compatibility-only aggregate permissions are now input-only across member/template snapshots and frontend action groups. Backend canonical snapshots expand `database.data_edit`, `database.ai_query`, `file.upload_create`, and `file.move_create` without retaining the old codes; frontend database/file direct pages and buttons now use exact `record.*`, `ai_query.*`, `file.upload`, and folder/file action permissions.
- 2026-06-29: File upload backend authorization now checks `file.upload` instead of legacy `file.upload_create`. File/dashboard visible permission helpers request fine-grained file permissions while still allowing old `file.upload_create` or `file.move_create` grants through compatibility expansion.
- 2026-06-29: Data-library knowledge-base file-reference routes stopped using legacy `knowledge_base.manage`. Candidate listing and ref creation now use `knowledge_base.document.create`, ref listing uses `knowledge_base.document.view`, embedding/retry task actions use `knowledge_base.index.manage`, and removal uses `knowledge_base.document.delete`.
- 2026-06-29: Workflow batch-test frontend routing now matches backend action split. Case library requires `workflow.view`; batch/result direct pages require `workflow.logs.view`; creating and immediately executing a batch requires `workflow.view`, `workflow.logs.view`, `workflow.update`, and `workflow.debug`. Batch-test buttons now hide update-only actions behind `workflow.update`, generation/retest/start actions behind `workflow.debug`, and cancellation behind `workflow.run.stop`.
- 2026-06-29: Workspace governance audit confirmed the intended owner/admin model is still the authority boundary. Frontend workspace member/settings and scheduled-task mutation actions use manager state (`organization owner/admin` or `workspace owner/admin`), while backend member/permission/task mutations require workspace governance permissions that are granted only through owner/admin authority. `workspace.view` and `workspace.member.view` remain membership permissions, not selectable ordinary template permissions.
- 2026-06-29: Default workspace role template audit confirmed Admin still receives all asset actions through compatibility-expanded snapshots, Advanced receives every configurable asset action except governance, Basic focuses on agent/workflow creation-editing plus supporting asset reads, and Read-only keeps inspection/debug permissions without mutation.
- 2026-06-29: Prompt/content-parse/dashboard retired-permission audit confirmed no ordinary member template or frontend permission module exposes `prompt.*`, `content_parse.*`, or `dashboard.*`. Prompt and content-parse APIs require current workspace membership via `workspace.view`; dashboard overview remains organization/workspace-context derived rather than permission-code gated.
- 2026-06-29: App-center audit confirmed runnable app lists and console overview call `useRunnableWebApps({ workspaceId: null })`, detail pages wait for authorized runnable membership before loading WebApp config, and backend organization-scope candidate lookup is separated from audience workspace matching so account/department/organization grants still work for users without joined workspaces.
- 2026-06-29: Workspace settings access decision recorded: the route can remain reachable to joined members for basic info/self-leave, but all governance mutations stay owner/admin gated and no ordinary `workspace.*` permission is reintroduced.
- 2026-06-29: File UI stopped directly consuming legacy `file.upload_create`, `file.move_create`, and broad file-manage groups. Upload, folder management, preview, download, update, delete, and processing actions now use exact action groups.
- 2026-06-29: Added direct-page gates for database operation logs, BI search, table records, table schema, AI table creation, table data import, table prompt editing, dataset documents, graph, hit-testing, batch-testing, and file detail. File detail backend now uses readable file permissions instead of download-only authorization.
- 2026-06-29: Agent/workflow UI and backend helpers branch action permissions by runtime type.
- 2026-06-29: Dashboard, database, knowledge base, file, and app-center helpers were aligned so create-only or published-access users do not lose their entry points.
