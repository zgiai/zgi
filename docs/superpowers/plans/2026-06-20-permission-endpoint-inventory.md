# Permission Endpoint Inventory

Date: 2026-06-20
Status: MVP closure evidence in progress

This inventory records the expected authorization boundary for the permission foundation MVP. It is intentionally focused on the routes and surfaces involved in the product-user versus builder split.

## Scope Legend

- `auth`: authenticated account only.
- `org_member`: active member of the current organization.
- `org_admin`: organization owner/admin.
- `workspace_perm`: concrete workspace plus a `WorkspacePermissionCode`.
- `published_runtime`: published surface grant; not the same as builder workspace permission.
- `public_runtime`: intentionally public published surface.
- `internal_runtime`: internal platform invocation; should not be disabled by webapp/API exposure settings.

Required invariant: frontend workspace selection is only a convenience. Every backend detail, mutation, preview, and subresource endpoint must independently validate organization, workspace, account, and required permission before returning data or changing state.

## Publication Surfaces

| Surface | Current Fields Or Routes | MVP Boundary |
| --- | --- | --- |
| Web app | `agents.web_app_status`, `PATCH /console/api/agents/:agent_id/webapp/status` | `web_app_enabled` plus public grant in the MVP. Existing active status maps to enabled during migration; backend writes reject non-public webapp grants until the public config/auth handshake decision below is made. |
| API service | `agents.enable_api`, `agent_api_keys`, `/api/v1/*` external routes | `api_service_enabled` is checked after API key validation and before external workflow/chat execution. Existing valid keys remain compatible when the surface is enabled; API key runtime calls use the API key's agent/workspace scope and do not require the API key ID to be a workspace member. API key management routes resolve the route agent's workspace for builder authorization and repository scope instead of treating the current organization ID as the key tenant. External workflow/chat file inputs require `upload_file_id` ownership by the current API key before extraction or execution. External run detail and task stop use the API key's workspace/agent/key scope and the workflow run service's run ownership check before returning detail or cancelling. Backend writes reject non-public API grants until API key audience semantics are decided. |
| Built-in app | `/console/api/built-in-workflows`, `/console/api/built-in-workflows/:scenario/runtime-surfaces`, app-center `/console/api/agents/runnable-webapps`, `published_runtime_surface_grants` | `builtin_app_enabled` plus org/department/account grants. MVP catalog route has authenticated organization context without a workspace prerequisite, defaults to visible for compatibility when no rows exist, and now evaluates built-in workflow candidates through the compatibility-preserving runtimeauth candidate batch filter. App-center evaluates agent `builtin_app` grants through the same batch filter. Both list paths load department audience once per request; direct built-in detail still repeats single-resource authorization. Organization admins can read/write `builtin_workflow` surface rows through the management API. Account/department grant writes must target active subjects in the current organization. |
| Internal invocation | workflow bindings, automation, scheduled tasks | `internal_runtime`; based on published availability, not webapp/API exposure. Backend writes reject disabled or non-internal grants for this surface. |

`GET /console/api/account/capabilities` is the frontend contract for these surfaces. It keeps the legacy `enabled/mode` shape and now also returns `grant_subject_types` per surface plus `runtime_audience.subject_types` for the current account, including active department audience when the member belongs to departments. The surface contract is explicit and regression-covered: `webapp` and `api` expose only `public` subjects until their audience handshake is decided, `builtin_app` exposes `organization`, `department`, and `account`, and `internal` exposes only `internal`. The same response also declares `runtime_resource_lists` metadata for the dedicated app-center and built-in workflow catalog list endpoints: `app_center` maps to `agent` + `builtin_app` at `/console/api/agents/runnable-webapps`, and `built_in_workflows` maps to `builtin_workflow` + `builtin_app` at `/console/api/built-in-workflows`, both using `runtimeauth_candidate_filter`. Frontend resource-list hooks consume these gates before querying or returning cached list data. No-organization context responses keep the same metadata with `enabled=false` rather than omitting the contract.

Minimal storage contract:

- `published_runtime_surfaces`: one active row per `resource_type`, `resource_id`, and `surface`, scoped by organization and optional workspace.
- `published_runtime_surface_grants`: subject rows for `public`, `organization`, `department`, `account`, or `internal`.
- `runtimeauth.Store.ListAuthorizedResourceIDs`: reusable persisted-resource index primitive for future resource-specific app/workflow lists. It evaluates enabled persisted surface rows with the same `SurfaceAuthorization.Allows` semantics used by single-resource checks; legacy fallback candidate selection remains owned by each caller because fallback state can live outside `published_runtime_surfaces`.
- `runtimeauth.Store.FilterAuthorizedResourceIDs`: compatibility-preserving batch filter for caller-owned candidate resources. It overlays persisted surface rows over each candidate's fallback policy, then evaluates the same `SurfaceAuthorization.Allows` semantics; this is the safer next integration point for app-center and built-in catalog lists because it does not drop no-row fallback or legacy seeded `builtin_app` compatibility.
- Account capabilities `runtime_resource_lists`: descriptive frontend discovery contract for the app-center and built-in catalog list endpoints. Frontend hooks gate list queries and cached results on this metadata, but the response does not inline authorized resource IDs, so resource candidate ownership and fallback compatibility stay inside the dedicated app/workflow services.
- Agent migration seed maps `web_app_status` to `webapp`, `enable_api` to `api`, `internal=true`, and a legacy `builtin_app` placeholder. For compatibility, an active legacy webapp with `builtin_app=false`, `compatibility_source=legacy_agent_fields`, and no grants is treated as app-center visible until a builder saves an explicit `builtin_app` grant policy.

Decision needed before enabling webapp/API account or department grants:

- Next-phase review brief: `docs/superpowers/plans/2026-06-21-runtime-authorization-next-phase.md`.

- Webapp config currently lives on public `/console/api/webapps/:web_app_id/config` with `SetupRequired` only. Runtime calls under `/webapps/:web_app_id/*` have `WebAppAuthMiddleware` and can build an account audience, but the config endpoint cannot safely evaluate account/department grants without either optional auth parsing or a separate public capability response. Directly persisting non-public webapp grants now would make config visibility and runtime execution disagree.
- Webapp user migration currently lives on `POST /console/api/workflows/migrate-user` without a route `web_app_id`. If webapp audience restrictions become non-public, migration needs either a resource-scoped migration route, a virtual-ID ownership proof, or an explicit decision to keep global virtual-account migration compatibility.
- API key runtime validation currently knows the agent and workspace from the key, but it has no caller account or department audience. Supporting account/department grants for API requires a product choice: treat API keys as public bearer capabilities, bind keys to an owner audience, or require a caller/user claim on each API request.

Decision record to resolve before changing the current locked contract:

| Area | Option | Compatibility Impact | Required Technical Work |
| --- | --- | --- | --- |
| Webapp audience | Keep `webapp` public-only and use `builtin_app` grants only for app-center visibility | Preserves current public config, anonymous visitor, and runtime behavior; does not deliver private webapps | No contract change; keep rejecting non-public webapp grants and keep capabilities `grant_subject_types=["public"]` |
| Webapp audience | Add optional auth to the public config path and evaluate grants when identity exists | Keeps the route shape but introduces different config visibility for anonymous vs authenticated callers | Optional JWT/webapp identity parsing on config, shared `runtimeauth` audience evaluation for config/run/upload, frontend unavailable/auth-required states, browser regression for anonymous/private webapp |
| Webapp audience | Add a protected/gated config capability flow while public config returns only shell/offline metadata | Cleanly separates anonymous shell from private runtime config, but requires frontend route changes and migration plan | New protected config or capability endpoint behind `WebAppAuthMiddleware`, frontend gated config loading, compatibility fallback for existing public webapps |
| Webapp user migration | Keep global `/workflows/migrate-user` compatibility | Preserves current anonymous-to-authenticated handoff but cannot enforce resource-specific webapp audience | Document as intentionally global and keep only virtual/authenticated account proof |
| Webapp user migration | Add `/:web_app_id/migrate-user` with surface/audience checks | Enables private webapp migration consistency, but changes callers and must keep old route compatible during transition | New resource-scoped route, webapp audience check, dual-route migration period, frontend migration caller update |
| Webapp user migration | Require virtual ID ownership proof on the existing route | Avoids route change but adds proof format and replay/expiry semantics | Signed virtual-account proof or short-lived migration token, storage/validation, replay tests |
| API audience | Treat API keys as public bearer capabilities within the enabled API surface | Preserves current API key semantics and external integrations; account/department grants remain unsupported for API | No contract change; keep rejecting non-public API grants and keep capabilities `grant_subject_types=["public"]` |
| API audience | Bind API keys to owner account/department audience | Gives coarse account/department control without changing each request, but department changes and shared keys become policy-sensitive | Store/evaluate key owner audience, department lookup at request time or snapshot decision, tests for owner transfer and department removal |
| API audience | Require caller/user claim per API request | Strongest user-level API authorization, but breaks simple bearer-key integrations without new caller token flow | Request claim format, signing/validation, SDK/docs changes, runtimeauth audience from caller claim, migration strategy for existing keys |

Until those rows are decided, the current implementation intentionally keeps `webapp` and `api` grant subjects public-only at capabilities, frontend management UI, management API, and storage boundaries.

## Current High-Risk Gaps

- Workspace permission filter routes accepted `type=create_agent|create_database|create_knowledge` but did not enforce the corresponding permission code. This is now covered by a targeted service regression.
- Workspace self-routes now resolve the route workspace's real organization before permission checks. Focused handler regressions cover statistics denial before service calls without `workspace.view` and workspace update by organization admin/owner users who have no `workspace_members` row for that workspace.
- Workspace member legacy routes now route their `workspace.view`/`workspace.manage` checks through a helper that resolves the workspace's real organization first. Focused regressions cover current-workspace member listing, route workspace member-extension listing, and cancel-invite denial before target member lookup.
- Organization workspace management routes now reject cross-organization route workspace IDs before permission checks or target account/workspace lookups. Focused regressions cover workspace assets, member removal, member role updates, and batch member add routes.
- Legacy `OrganizationService.CheckWorkspacePermission` and `CheckWorkspaceOrganizationAnyPermission` now fail closed for empty or missing organization scope. Handler routes still resolve workspace organization scope before calling them to keep cross-organization workspace IDs rejected before lower-level service work.
- Account context repair can repopulate `current_workspace_id`; this prevents a stable organization-only mode.
- Account context now exposes an explicit `mode` API and no longer needs JSON null/empty-field ambiguity to enter organization-only mode. Docker zgi-c and local headless Chrome regressions now cover organization-mode product pages, the workspace switcher organization-mode entry, mobile product/asset route smoke, workspace-required asset pages, and temporary switch-back-to-workspace recovery. Route-access regression also locks the switcher entry, organization-mode redirect, context API mode, capabilities refresh, and cross-tab context broadcast; remaining browser debt is cross-browser/manual visual polish for the switcher popover rather than functional coverage.
- Account context service/handler regressions now cover the no-workspace contract used after login: profile/context responses keep `current_workspace_id=null`, `GET /account/context` derives `mode=organization`, and `EnsureCurrentOrganizationID` returns the organization ID without selecting a workspace.
- Account capabilities now give the frontend a backend contract for organization-scope and workspace-scope route decisions. Handler regressions lock the HTTP JSON shape for no-workspace organization mode: organization product surfaces stay enabled, workspace permissions stay empty with `workspace_required=true`, runtime audience includes organization/account, and all four runtime surface metadata entries are present. Shared console route metadata defines the organization-scoped shell/work entries and is covered by route-access regressions, so chat, image, app, app detail, settings, and workspace asset boundaries stay aligned. - Asset list routes tend to filter by workspace, while detail and subresource routes often load by bare ID.
- Dashboard recent-work is now workspace scoped: the handler requires a non-empty current account-context workspace plus `workspace.view` before calling the service, and the service filters agents, datasets, conversations, and data sources to that workspace instead of returning organization-wide asset names and IDs.
